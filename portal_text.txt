Философия дизайна
Сахалин — это географическая и культурная точка слияния:
    Японское наследие — минимализм, асимметрия, «ма» (пространство между объектами), природные материалы, ваби-саби (красота неидеальности)
    Корейское влияние — смелая типографика, контрастные цветовые блоки, геометрический порядок, цифровая эстетика
    Русское присутствие — функциональность, информационная плотность, суровая эстетика Севера
    Азиатский контекст — вертикальная типографика как декоративный элемент, иероглифические акценты, натуральные текстуры

Типографика (2026: Typography as Architecture)
    Заголовки: Neo-Serif с характером — например, Fraunces или Playfair Display с увеличенной контрастностью. Крупные, viewport-scaled (clamp()), иногда с вертикальным написанием для декоративных элементов (отсылка к японской вертикали).
    Навигация/UI: Гротеск с моноширинными элементами — Inter + JetBrains Mono для метаданных, дат, тегов. Чистая инженерная эстетика.
    Акценты: Возможность использования корейского шрифта Noto Sans KR для декоративных вставок (названия районов, культурных рубрик).
Текстуры и материалы (2026: Tactile Brutalism)
    CSS-шум/зернистость: Лёгкий overlay noise.svg на фоне — имитация вулканического песка, старой фотобумаги, рисовой бумаги «ваши»
    1px solid borders: Для разделения карточек — не тени, а жёсткие линии в цвете #1E293B
    Фотографии: Обработка с плёночной зернистостью, приглушённые тона, высокий контраст — как японская фотография Дайсуке Йокоты
Композиция и сетка
    Bento Grid для главной: информационные блоки разного размера, как корейские модульные сетки, но с японским чувством пропорций
    Асимметрия: Смещение основного контента влево, декоративные вертикальные полосы справа — отсылка к японским макетам
    Щедрое пространство: «Ма» — пустое пространство как активный элемент дизайна, не дефицит
UI-элементы
    Кнопки: 0px border-radius (brutalism) или полностью pill-shaped — ничего промежуточного. 1px border, без теней.
    Карточки новостей: Чёткие прямоугольники, hover-эффект — не подъём тенью, а сдвиг вправо на 2px + изменение border-color на акцентный.
    Навигация: Фиксированная, минималистичная, с моноширинными индикаторами активного раздела.
Микро-интеракции (2026: Purposeful Motion)
    Появление текста: Посимвольное или пословное появление заголовков при скролле — как набор текста на пишущей машинке (отсылка к японской поэзии, медитативность)
    Переключение тем: Morning/Evening/Focus modes — утренний (светлый, тёплый), вечерний (тёмный, янтарный), фокусный (монохром, минимум анимации)
    Погода: Анимированные иконки в стиле «живые чернила» — минималистичные, но органичные

3. Типографическая система
Display / H1: Fraunces или Playfair Display — neo-serif с высоким контрастом. Размер: clamp(2.5rem, 5vw, 4.5rem). Вертикальное написание для декоративных маркеров секций.
H2-H4: Inter — чистый гротеск. clamp(1.25rem, 2vw, 1.75rem).
Body: Inter 16px/1.6.
UI / Meta / Data: JetBrains Mono — даты, теги, погода, курсы. 12px/1.4.
Корейские акценты: Noto Sans KR для названий районов, культурных рубрик (декоративно, не для основного текста).
4. Текстуры и материалы
CSS Noise Overlay: SVG-фильтр с 3% opacity на фоне — имитация вулканического песка и рисовой бумаги. Zero performance cost.
Фото-обработка: Пресет «Sakhalin Film» — повышенный контраст, desaturated colors (кроме неба), добавленная зернистость, лёгкий vignette. Как фотографии Дайсуке Йокоты.
Никаких градиентных теней: Глубина через layering и border, не через box-shadow.

8. Микро-интеракции
Появление заголовков: IntersectionObserver + посимвольное reveal через opacity + translateY(10px), задержка между символами 0.02s — эффект пишущей машинки, но плавный.
Скролл-индикатор: Тонкая линия #00E5CC сверху, ширина = прогресс скролла.
Hover на ссылках: Подчёркивание растёт слева направо (scaleX от 0 до 1), не мгновенное.
Переключение режимов: Cross-fade 0.4s, без резких скачков.
9. Machine Experience (MX) — для AI-агентов
Семантическая HTML-структура: <article> с datetime, <section> с aria-label, строгая иерархия H1→H6
Schema.org разметка: NewsArticle, Organization, Place (для локаций Сахалина)
Дизайн-токены: semantic tokens вместо literal (--color-accent-primary вместо --color-cyan-500)
Это критично для visibility в ChatGPT, Perplexity, Gemini — пользователи всё чаще спрашивают AI о новостях региона

2. Система комментариев
Модель данных (иерархия)
Комментарий:
├── id, news_id, user_id
├── parent_id (NULL = корневой, != NULL = ответ)
├── content (текст, markdown-разметка: жирный, курсив, ссылки)
├── status: pending / approved / rejected / deleted_by_user / deleted_by_moderator
├── created_at, updated_at, deleted_at (soft delete)
├── likes_count, dislikes_count
├── is_pinned (закреплён редактором)
└── ip_address, user_agent (для антиспама)
Жизненный цикл комментария
Пользователь пишет ──▶ Система проверяет ──▶ Решение
                         │
                         ├── Авто-модерация (ML/правила)
                         │   ├── Чёрный список слов ──▶ rejected
                         │   ├── Спам-фильтр (Akismet/свой) ──▶ pending
                         │   ├── Флуд-контроль (>10 комм/час) ──▶ pending
                         │   └── ОК ──▶ approved (если у пользователя репутация > 50)
                         │
                         └── Ручная модерация (если pending)
                             ├── Модератор одобряет ──▶ approved
                             └── Модератор отклоняет ──▶ rejected (с причиной)

Алгоритм «Утренний дайджест»:
1. Берём новости за последние 12 часов (19:00–07:00)
2. Фильтруем по подпискам пользователя (рубрики, теги)
3. Сортируем по весу:
   weight = просмотры * 0.3 + комментарии * 0.4 + время на странице * 0.2 + редакторский выбор * 0.1
4. Берём топ-5
5. Добавляем погоду (город по профилю, иначе Южно-Сахалинск)

8. Отправляем через очередь (batch: 1000 писем/минуту, чтобы не попасть в спам)

РАНЖИРОВАНИЕ (Scoring)
Score = w₁·ContentScore + w₂·UserScore + w₃·PopularityScore + w₄·FreshnessScore + w₅·DiversityScore + w₆·EditorialScore +w₇·ContextScore + w₈·BusinessScore
Веса адаптивные: персонализируются под пользователя (коллаборативный 
фильтр на уровне весов: если пользователь часто кликает на «свежие» новости, w₄ растёт для него)
ВЫХОД: Персонализированная лента
(50 элементов → фильтрация → 20 → ранжирование → 10)
1. Компоненты Score
1.1 ContentScore (контентная близость)
TF-IDF + Word2Vec по тексту новости
Шаг 1: Векторизуем профиль пользователя
- Берём все просмотренные новости за последние 90 дней
- Складываем TF-IDF вектора, взвешенные временем (recency decay)
- Получаем UserContentVector: 300-мерный вектор
Шаг 2: Векторизуем кандидата
- TF-IDF + Word2Vec заголовка, lead, тегов
- Получаем CandidateVector
Шаг 3: ContentScore = cosine_similarity(UserContentVector, CandidateVector)
- Нормализуем: 0.0 → 1.0
Рубрики и теги (категориальные):
Если пользователь подписан на рубрику:
  ContentScore += 0.15
Если тег новости есть в подписках пользователя:
  ContentScore += 0.05 за каждый совпадающий тег (макс 0.25)
Если автор в подписках:
  ContentScore += 0.10
1.2 UserScore (коллаборативная фильтрация)
User-User Collaborative Filtering (для пользователей с историей >20 просмотров)
Шаг 1: Найти «похожих» пользователей
- Для активного пользователя U: берём его вектор взаимодействий
  (item_id × action_type: view=1, like=3, share=5, comment=4, save=4)
- Считаем cosine similarity со всеми пользователями
- Берём топ-50 nearest neighbors
Шаг 2: Рекомендовать то, что смотрели соседи, но U не видел
- CandidateScore = weighted_average(neighbor_ratings, similarity)
- Нормализуем: 0.0 → 1.0
Для новых пользователей (<20 просмотров):
  → Fallback на ContentScore + PopularityScore + ContextScore
Item-Item Collaborative Filtering (альтернатива для масштабирования):
- Предвычисляем матрицу похожести новостей (cosine similarity по взаимодействиям)
- Если пользователь смотрел новость A → рекомендовать похожие B, C, D
- Эффективнее для больших баз (O(n_items²) предвычисляется offline)
1.3 PopularityScore (популярность)
PopularityScore = (
  0.4 × log(просмотры + 1) / max_log_views +
  0.3 × (комментарии / max_comments) +
  0.2 × (поделились / max_shares) +
  0.1 × (среднее время чтения / max_read_time)
)
Нормализация: делим на максимум за последние 7 дней
Пенализация за «вирусный шлак»:
- Если просмотры высокие, но время чтения <15 сек → PopularityScore *= 0.3
- Если комментариев мало относительно просмотров → штраф
1.4 FreshnessScore (свежесть)
Функция затухания (decay):
t = часы с момента публикации
Если t < 2:    FreshnessScore = 1.0
Если t < 6:    FreshnessScore = 0.9
Если t < 12:   FreshnessScore = 0.75
Если t < 24:   FreshnessScore = 0.6
Если t < 48:   FreshnessScore = 0.45
Если t < 72:   FreshnessScore = 0.35
Если t < 168:  FreshnessScore = 0.25  (7 дней)
Если t < 720:  FreshnessScore = 0.15  (30 дней)
Иначе:         FreshnessScore = 0.05  (архив)
Особые случаи:
- «Тема дня» (editorial pick): FreshnessScore = 1.0 на 24 часа
- Экстренная новость: FreshnessScore = 1.0 на 6 часов
- «Вечное» (справочный контент): FreshnessScore фиксирован = 0.3
1.5 DiversityScore (разнообразие)
Проблема: Алгоритм может «застрять» в одной рубрике (filter bubble)
DiversityScore для кандидата C в контексте уже выбранных элементов S:
1. Рубрика: если в S уже есть 2+ новости той же рубрики → DiversityScore *= 0.5
2. Автор: если в S уже есть новость этого автора → DiversityScore *= 0.7
3. Теги: Jaccard similarity тегов C с S < 0.3 → DiversityScore *= 1.2 (поощрение разнообразия)
4. Медиа-тип: если в S все текстовые, а C — видео → DiversityScore *= 1.15
Финальный DiversityScore: 0.0 → 1.0
Алгоритм MMR (Maximal Marginal Relevance) для финальной выборки:
MMR = λ × Relevance(C) − (1−λ) × max[Similarity(C, Sᵢ)]
λ = 0.7 (баланс релевантности и разнообразия)
- Для «консервативных» пользователей: λ = 0.8
- Для «открытых» пользователей: λ = 0.6

1.8 BusinessScore (бизнес-логика)
Платные продвижения объявлений:
├── VIP-объявление → BusinessScore = 0.9 (гарантированно в ленте, но не топ-1)
├── Выделенное → BusinessScore = 0.7
└── Поднятое → BusinessScore = 0.5
Рекламные интеграции (нативные):
├── Помечены «Реклама» → BusinessScore = 0.6
└── Ограничение: не более 1 рекламной на 10 органических
Премиум-пользователи (Sakhcom+):
├── BusinessScore для рекламы = 0 (не показываем рекламу)
└── BusinessScore для «эксклюзивных» материалов = 1.2
2. Финальная формула ранжирования
Для каждого кандидата C:
BaseScore = (
  0.25 × ContentScore +
  0.20 × UserScore +
  0.15 × PopularityScore +
  0.15 × FreshnessScore +
  0.10 × DiversityScore +
  0.10 × EditorialScore +
  0.03 × ContextScore +
  0.02 × BusinessScore
)
# Адаптивные веса (персонализация весов):
Если пользователь «новостной наркоман» (>50 просмотров/день):
  FreshnessScore вес *= 1.5, PopularityScore вес *= 0.7
Если пользователь «глубокий читатель» (среднее время >5 мин):
  ContentScore вес *= 1.3, PopularityScore вес *= 0.6
Если пользователь «социальный» (много комментариев):
  PopularityScore вес *= 1.2, UserScore вес *= 1.2
# Нормализация весов (чтобы сумма = 1.0)
FinalScore = BaseScore × DiversityPenalty
Где DiversityPenalty:
  Если рубрика C уже представлена в выборке N раз:
    penalty = 0.8^N
  Если автор C уже представлен:
    penalty *= 0.9
3. Пайплайн формирования ленты
Шаг 1: Кандидатская выборка (Candidate Generation)
├── Берём все новости за последние 7 дней
├── + «вечные» материалы (справочники, гиды)
├── + архивные, если пользователь искал похожее
├── Фильтруем: status = published, не скрыта для этого пользователя
├── Получаем ~500–2000 кандидатов
└── Время: < 50 мс (индекс Elasticsearch)
Шаг 2: Предфильтрация (Pre-filtering)
├── Исключаем: уже просмотренные пользователем (кроме «вернуться»)
├── Исключаем: из блэклиста рубрик (пользователь отписался)
├── Исключаем: возрастные ограничения (если профиль <18)
├── Оставляем ~200 кандидатов
└── Время: < 20 мс
Шаг 3: Скоринг (Scoring)
├── Для каждого кандидата считаем все 8 компонентов
├── Параллельно, batch processing
├── Получаем score для 200 кандидатов
└── Время: < 100 мс (Redis для кэша, предвычисленные векторы)
Шаг 4: Ранжирование и диверсификация (Ranking)
├── Сортируем по FinalScore
├── Применяем MMR для топ-50
├── Проверяем: не более 3 одинаковых рубрик подряд
├── Проверяем: не более 1 видео подряд (для баланса медиа)
├── Проверяем: editorial picks в правильных позициях
├── Получаем ранжированный список 50
└── Время: < 30 мс
Шаг 5: Постраничная выдача (Pagination)
├── Страница 1: элементы 1–10
├── Страница 2: элементы 11–20 (при скролле/клике «ещё»)
├── ...
└── Кэшируем в Redis: key = user_id + page + context_hash, TTL = 5 минут

Обучение весов (Online Learning)
Каждые 24 часа:
1. Собираем логи взаимодействий за день
2. Считаем метрику: nDCG@10 (Normalized Discounted Cumulative Gain)
   - Идеальная лента: все кликнутые элементы наверху
3. Если nDCG < 0.6 → триггерим переобучение
4. A/B тест: 10% пользователей на новые веса, 90% на старые
5. Если новые веса дают +5% CTR → раскатываем на 100%

7. Этика и прозрачность
1. Объяснимость:
   - При наведении на карточку: «Рекомендовано, потому что вы читали...»
   - В настройках: «Вот что мы знаем о ваших интересах» (визуализация профиля)
2. Контроль:
   - «Не показывать мне [рубрику]» → мгновенно в блэклист
   - «Сбросить мои рекомендации» → очистка профиля, холодный старт
   - «Только хронология» → режим без алгоритма
3. Разнообразие:
   - Гарантия: минимум 20% материалов из рубрик, на которые не подписан
   - «Случайная новость» кнопка — пробивает filter bubble
4. Редакционная независимость:
   - Editorial pick не может быть «забит» алгоритмом
   - Важные, но непопулярные темы (экология, культура) получают boost

13. Этика и прозрачность поиска
1. Объяснимость ранжирования:
   - При наведении: «Почему этот результат?»
     • «Совпадение в заголовке»
     • «Популярный материал»
     • «Рядом с вами»
     • «Похоже на то, что вы искали раньше»
2. Контроль:
   - «Искать точное совпадение» (отключить синонимы)
   - «Искать без персонализации» (инкогнито-режим)
   - «Сортировать по дате» (override релевантности)
3. Нейтральность:
   - Editorial boost не применяется в поиске (только в ленте)
   - Реклама маркируется и не бустится в organic search
   - Политические материалы ранжируются по тем же правилам
4. Приватность:
   - История поиска хранится 90 дней
   - Пользователь может удалить историю
   - Нет передачи поисковых данных третьим лицам
